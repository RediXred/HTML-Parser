Проблема

В твоем текущем лексеме (lexer.l) все имена тегов, атрибуты и текст вне тегов обрабатываются одним правилом:

c
{TEXT} { yylval.str = strdup(yytext); TOKEN(TEXT); }

Где TEXT определен как:

c
TEXT ([^<>\n\"\'= \t\r\f\v\a\/]|{NEWLINE})+

Это регулярное выражение захватывает любую последовательность символов, которая не включает <, >, кавычки, пробельные символы и некоторые другие. В результате для HTML вроде <div class="container">Hello</div> токены получаются такими:

  • < → OPENING_TAG_BRACKET
  • div → TEXT
  • class → TEXT
  • = → ASSIGNMENT
  • "container" → DOUBLE_QUOTE_STRING
  • > → CLOSING_TAG_BRACKET
  • Hello → TEXT
  • < → OPENING_TAG_BRACKET
  • / → SLASH
  • div → TEXT
  • > → CLOSING_TAG_BRACKET

Парсер в gram.y справляется с этим, потому что он различает, где TEXT — это имя тега (после <), а где — атрибут (в списке атрибутов). Но ты хочешь, чтобы лексер сам выдавал TAG_NAME для имен тегов (например, div), ATTR_NAME для имен атрибутов (например, class) и TEXT для содержимого вне тегов (например, Hello). Когда ты попробовал это сделать, добавив отдельные правила вроде:

c
[a-zA-Z][a-zA-Z0-9]* { TOKEN(TAG_NAME); } [a-zA-Z][a-zA-Z0-9]* { TOKEN(ATTR_NAME); } [^<]+ { TOKEN(TEXT); }

Flex не мог понять, какое правило применять, так как паттерны пересекаются, и выбирал либо первое определенное правило, либо самое длинное совпадение, что ломало логику парсинга.

Почему возникает конфликт?

Flex — это лексический анализатор, который работает на основе регулярных выражений и не учитывает контекст (например, находится ли он внутри тега или снаружи). Если у TAG_NAME, ATTR_NAME и TEXT похожие или одинаковые паттерны (например, [a-zA-Z][a-zA-Z0-9]*), Flex не знает, какой токен вернуть для строки вроде div или class, и использует правило, определенное раньше в файле. Это и вызывает конфликт.

Решение

Чтобы разделить TAG_NAME, ATTR_NAME и TEXT, нужно дать Flex возможность учитывать контекст. Самый эффективный способ — использовать состояния (start conditions). Состояния позволяют переключать наборы правил в зависимости от того, где мы находимся в HTML: внутри тега, в атрибутах или вне тегов. Давай реализуем это шаг за шагом.

1. Определим состояния

Добавим в секцию определений lexer.l следующие состояния:

c
%x INTAG /* Внутри тега: после '<' и до '>' */ %x INTAG_CLOSE /* Для закрывающих тегов: после '</' */ %x INATTR /* Для атрибутов: после имени тега внутри '<...>' */
  • INITIAL — начальное состояние, когда мы вне тегов.
  • INTAG — состояние для разбора открывающих тегов.
  • INTAG_CLOSE — состояние для закрывающих тегов.
  • INATTR — состояние для атрибутов после имени тега.

2. Обновим грамматику

Твоя грамматика уже объявляет токены TAG_NAME и ATTR_NAME, но использует TEXT. Изменим правила в gram.y, чтобы использовать TAG_NAME для имен тегов и ATTR_NAME для имен атрибутов:

В чем проблема?

В HTML символ < может означать две вещи:

  • Начало тега, например <p> или </p>.
  • Просто символ в тексте, как в твоем случае x<y.

Когда лексер встречает <, он обычно думает, что это начало тега, и переключается в состояние INTAG (состояние для обработки тегов). В примере <p>Hi world, x<y</p>:

  • <p> правильно распознается как тег.
  • Но при x<y лексер тоже видит < и переходит в INTAG, хотя x<y — это не тег, а обычный текст внутри параграфа.

Это происходит потому, что текущая логика лексера не отличает < как часть текста от < как начала тега. В результате он ошибочно интерпретирует x<y как попытку открыть тег, что ломает разбор.

Почему так происходит?

В HTML < и > допустимы внутри текста (например, в математических выражениях вроде x<y или при описании кода). Браузеры и парсеры должны уметь определять, когда < начинает тег, а когда нет. Обычно:

  • Если после < идет буква (a-z, A-Z), это считается началом тега (например, <div>).
  • Если после < идет /, это закрывающий тег (например, </div>).
  • В других случаях < может быть просто текстом.

В твоем примере после < идет y, что похоже на начало тега, но в контексте это не тег, а текст. Текущий лексер этого не учитывает и "слепо" переходит в INTAG.

Как исправить?

Чтобы лексер правильно обрабатывал такие случаи, нужно изменить его логику. Вот простое и рабочее решение:

Идея

Считать < началом тега только в определенных случаях:

  • Если за < идет буква (a-z, A-Z), переходим в INTAG (открывающий тег).
  • Если за < идет /, переходим в состояние для закрывающего тега.
  • В остальных случаях < остается частью текста.

Пример кода для лексера (на Flex)

c
<INITIAL>"<"[a-zA-Z] { unput(yytext[1]); // Вернуть букву обратно в поток BEGIN(INTAG); // Перейти в состояние тега return OPENING_TAG_BRACKET; } <INITIAL>"</" { BEGIN(INTAG_CLOSE); // Перейти в состояние закрывающего тега return OPENING_TAG_BRACKET; return SLASH; } <INITIAL>"<" { yylval.str = strdup("<"); // Сохранить "<" как текст return TEXT; }
  • "<"[a-zA-Z]: Если < followed by буква, это тег. Возвращаем < как OPENING_TAG_BRACKET и переходим в INTAG.
  • "</": Если < followed by /, это закрывающий тег. Переходим в соответствующее состояние.
  • "<": В любом другом случае < — это текст, возвращаем его как TEXT.

Как это работает на твоем примере?

Возьмем <p>Hi world, x<y</p>:

  1. <p>:
    • < followed by p (буква) → OPENING_TAG_BRACKET, переходим в INTAG.
    • p → TAG_NAME.
    • > → CLOSING_TAG_BRACKET.
  2. Hi world, x<y:
    • До x<y все идет как TEXT.
    • При < в x<y: < не followed by / или буквой после проверки → возвращаем < как TEXT.
    • x<y целиком становится частью TEXT.
  3. </p>:
    • </ → OPENING_TAG_BRACKET и SLASH, переходим в INTAG_CLOSE.
    • p → TAG_NAME.
    • > → CLOSING_TAG_BRACKET.

Почему это решает проблему?

Теперь лексер не "путается" на < внутри текста, а смотрит на следующий символ, чтобы принять решение. Если < не соответствует шаблону тега, оно остается текстом. Это простой и эффективный способ для большинства случаев.

Что еще учесть?

  • Экранирование: В реальном HTML < в тексте часто пишут как &lt;. Если в твоем случае это не так, решение выше подойдет. Но если экранирование есть, нужно добавить обработку &lt; и &gt; как текста.
  • Невалидный HTML: Если документ содержит ошибки (например, <> или <123>), лексер все еще может их интерпретировать как текст, что безопасно, но может не совпадать с поведением браузеров.
  • Сложные случаи: Внутри <script> или <style> тегов < и > встречаются чаще, но для них обычно есть отдельное состояние вроде RAWTEXT.

Как правильно сделать?

Чтобы различать < как начало тега и < как часть текста, нужно учитывать контекст — находимся ли мы уже внутри тега или в текстовом содержимом между тегами. Лексер сам по себе не "помнит" структуру документа, но мы можем использовать состояния Flex, чтобы отслеживать это.

Идея

  • В состоянии INITIAL мы ожидаем либо начало тега, либо текст.
  • После открытия тега (например, <p>), мы переходим в состояние, где < внутри содержимого обрабатывается как текст, а не как тег, пока не встретим закрывающий тег.
  • Для этого добавим состояние INTEXT (содержимое между тегами).

Обновленные состояния

  • INITIAL: начальное состояние, где мы ждем теги или текст.
  • INTAG: внутри открывающего тега (<div ...>).
  • INTAG_CLOSE: внутри закрывающего тега (</div>).
  • INATTR: для атрибутов внутри тега.
  • INTEXT: для текста между тегами (например, Hi world, x<y).

    • Итог: Hi world, x<y разбивается на куски текста, а < не интерпретируется как тег.
  3. </p>:
    • </ → OPENING_TAG_BRACKET и SLASH, переходим в INTAG_CLOSE.
    • p → TAG_NAME.
    • > → CLOSING_TAG_BRACKET, возвращаемся в INITIAL.

Почему это работает?

  • В состоянии INTEXT мы явно говорим, что < — это текст, если только оно не часть </ (закрывающего тега).
  • Переход в INTAG происходит только в начале документа или после закрывающего тега, а не внутри содержимого.

Ограничения

  • Только для валидного HTML: Если есть вложенные < без закрытия (например, <p>x<y), это не сломает парсер, но <y будет текстом, что может быть не тем, чего ты хочешь в случае ошибок.
  • Экранирование: < и > в реальном HTML часто экранируются как < и >. Если тебе это нужно, добавь правила для их обработки.

Итог

Анализ секции <INITIAL>

Правило [^<>\n]+ означает "любой символ, кроме <, > и \n, один или более раз". Оно находится выше правила \n { } и перехватывает все символы, кроме <, > и \n. Почему предупреждение?

Flex выдает предупреждение, когда видит, что правило никогда не будет использовано. В данном случае \n { } не нужно, потому что:

  • \n уже обрабатывается глобальным правилом {NEWLINE} {/* Игнорируем новую строку */} в начале файла.
  • Оно избыточно в <INITIAL>, так как [^<>\n]+ и так исключает \n из текста.

int parse_result = yyparse(); как в лексере вернуть ошибку?

Чтобы в лексере (lexer.l) вернуть ошибку и прервать парсинг, нужно воздействовать на парсер (yyparse), который вызывается в функции main с кодом int parse_result = yyparse();. В Flex лексер сам по себе не "возвращает ошибку" напрямую в parse_result, но он может сигнализировать об ошибке парсеру через вызов функции yyerror и завершение работы лексера, что приведет к возврату ненулевого значения из yyparse (обычно 1), указывающего на неудачу.

Давай разберем, как это сделать.

Как работает yyparse и ошибки?

  • Функция yyparse (генерируемая Bison) возвращает:
    • 0, если парсинг успешен.
    • Ненулевое значение (обычно 1), если произошла ошибка (синтаксическая или другая).
  • Лексер (yylex) вызывается yyparse для получения следующего токена. Если лексер "ломается" или сообщает об ошибке, парсер может завершить работу.
  • Функция yyerror(const char *msg) используется для вывода сообщения об ошибке и вызывается автоматически при синтаксических ошибках в парсере или явно из лексера.

Как вернуть ошибку из лексера?

Есть несколько способов сообщить об ошибке из лексера и прервать парсинг:

1. Вызвать yyerror и вернуть 0

В лексере можно вызвать yyerror с сообщением об ошибке и вернуть специальный токен 0 (конец ввода), что заставит yyparse завершить работу:

c
<TAGSEC>{ . { yyerror("Unexpected character inside tag"); return 0; // Сигнал конца ввода } }
  • Что происходит?
    • yyerror("Unexpected character inside tag") выведет сообщение вроде Error at line X: Unexpected character inside tag.
    • return 0 сообщает парсеру, что ввод закончился, и yyparse завершится с ошибкой (обычно возвращает 1).
  • Результат: parse_result будет равно 1, что сигнализирует об ошибке.

Пример:

Если в <p>Hello< встретится < внутри TAGSEC, где это недопустимо:

c
<TAGSEC>{ "<" { yyerror("Nested '<' inside tag is not allowed"); return 0; } }

2. Использовать YYABORT в парсере через лексер

Если ты хочешь явно прервать парсинг с ошибкой, можно вернуть специальный код, который парсер интерпретирует как команду завершения. Для этого нужно:

  • Определить специальный токен ошибки в gram.y.
  • Вернуть его из лексера.

В gram.y добавь:

c
%token ERROR_TOKEN

В lexer.l:

c
<TAGSEC>{ . { yyerror("Unexpected character inside tag"); return ERROR_TOKEN; } }

В gram.y обработай этот токен:

c
htmlElement: ERROR_TOKEN { YYABORT; } /* остальные правила */
  • YYABORT: Макрос Bison, который немедленно прерывает yyparse и возвращает 1.
  • Результат: parse_result будет 1, и парсинг остановится.

3. Прямой выход через exit

Самый радикальный способ — завершить всю программу прямо из лексера:

c
<TAGSEC>{ . { yyerror("Unexpected character inside tag"); exit(1); // Немедленный выход с кодом ошибки } }
  • Плюс: Просто и сразу прерывает выполнение.
  • Минус: Не дает парсеру корректно завершить работу, и parse_result не будет установлен, так как программа просто завершится.

Рекомендация

Лучше всего использовать первый способ (вызов yyerror и return 0), так как он:

  • Совместим с текущей структурой твоего кода.
  • Позволяет парсеру корректно обработать ошибку и вернуть 1 в parse_result.
  • Не требует изменений в грамматике.

Пример для твоего случая:

c
<TAGSEC>{ . { char msg[256]; snprintf(msg, sizeof(msg), "Unexpected character inside tag: %s", yytext); yyerror(msg); return 0; } }

Вывод в консоли:

Если встретится неожиданный символ, например < внутри тега:

Error at line 3: Unexpected character inside tag: < Parsing failed

И parse_result будет равно 1.

Проверка в main

Твой текущий main уже проверяет результат yyparse:

Если лексер вызовет yyerror и вернет 0, то yyparse завершится с ошибкой, и parse_result будет 1, что выведет "Parsing failed".

Давай разберем твою проблему с лексером: ты хочешь, чтобы имена атрибутов внутри тегов обрабатывались как ATTR_NAME, но сейчас они всегда попадают под правило для TAG_NAME. Это происходит из-за того, как Flex выбирает правила и из-за пересечения регулярных выражений для TAG_NAME и ATTR_NAME в состоянии TAGSEC. Давай разберем причину и исправим это.

Почему ATTR_NAME не срабатывает?

Посмотрим на правила в состоянии TAGSEC:

c
<TAGSEC>{ [a-zA-Z][a-zA-Z0-9\-]* { yylval.str = strdup(yytext); return TAG_NAME; } "/>" { BEGIN(INITIAL); return TAG_SELF_CLOSE; } ">" { BEGIN(INITIAL); return CLOSING_TAG_BRACKET; } [a-zA-Z_:][a-zA-Z0-9_\-.:]* { printf("attr\n"); yylval.str = strdup(yytext); return ATTR_NAME; } "=" { return ASSIGNMENT; } "\""[^"]*"\"" { yylval.str = strdup(yytext); return ATTR_VALUE; } "'"[^']*"'" { yylval.str = strdup(yytext); return ATTR_VALUE; } [ \t\r\n]+ { /* пропускаем пробелы */ } . { yyerror("Unsupported characters in tag section."); return 0; } }

Проблема

  • Регулярное выражение для TAG_NAME: [a-zA-Z][a-zA-Z0-9\-]*
    • Соответствует любому слову, начинающемуся с буквы, за которой могут следовать буквы, цифры или дефисы (например, div, class, data-id).
  • Регулярное выражение для ATTR_NAME: [a-zA-Z_:][a-zA-Z0-9_\-.:]*
    • Соответствует словам, начинающимся с буквы, _ или :, за которыми могут следовать буквы, цифры, дефисы, подчеркивания, точки или двоеточия (например, class, data-id, xml:lang).
  • Пересечение: Оба правила перекрываются для большинства имен атрибутов (например, class, id, style), так как они начинаются с буквы и могут содержать символы, допустимые в обоих паттернах.
  • Порядок в Flex: Flex выбирает первое правило в файле, если несколько правил подходят для одного и того же ввода. Поскольку [a-zA-Z][a-zA-Z0-9\-]* (для TAG_NAME) стоит выше, оно всегда "побеждает", и до [a-zA-Z_:][a-zA-Z0-9_\-.:]* (для ATTR_NAME) дело не доходит.

Пример

Для <div class="container">:

  1. < → OPENING_TAG_BRACKET, переходим в TAGSEC.
  2. div → TAG_NAME (срабатывает [a-zA-Z][a-zA-Z0-9\-]*).
  3. class → TAG_NAME (снова срабатывает [a-zA-Z][a-zA-Z0-9\-]*, а не ATTR_NAME).
  4. = → ASSIGNMENT.
  5. "container" → ATTR_VALUE.
  6. > → CLOSING_TAG_BRACKET.

Ожидалось, что class станет ATTR_NAME, но оно стало TAG_NAME.

Как исправить?

Чтобы различать TAG_NAME и ATTR_NAME, нужно дать Flex контекстную информацию. Имя тега всегда идет первым после < или </, а атрибуты следуют за ним. Мы можем:

  1. Использовать отдельное состояние для атрибутов.
  2. Ограничить правило TAG_NAME так, чтобы оно срабатывало только для первого слова после <.

Решение: Добавить состояние INATTR

  • Состояние TAGSEC будет обрабатывать только имя тега.
  • После имени тега перейдем в состояние INATTR для атрибутов.

Что изменилось?

  1. Добавлено состояние INATTR:
    • После обработки TAG_NAME в TAGSEC лексер переходит в INATTR.
    • В INATTR обрабатываются атрибуты (ATTR_NAME), значения (ATTR_VALUE), и закрытие тега (> или />).
  2. Ограничена роль TAGSEC:
    • Теперь TAGSEC отвечает только за имя тега после < или </.
    • После имени тега сразу переходим в INATTR для дальнейшего разбора.
  3. Убрано пересечение:
    • TAG_NAME срабатывает только для первого слова после < или </.
    • ATTR_NAME обрабатывает последующие слова в теге.

Как это работает на <div class="container">?

  1. < → OPENING_TAG_BRACKET, переходим в TAGSEC.
  2. div → TAG_NAME, переходим в INATTR.
  3. class → ATTR_NAME (в INATTR).
  4. = → ASSIGNMENT.
  5. "container" → ATTR_VALUE.
  6. > → CLOSING_TAG_BRACKET, возвращаемся в INITIAL.
